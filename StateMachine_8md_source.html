3 ## What is a Finite State Machine (FSM)
4 
5 - A reactive system whose response to a particular stimulus (a signal, or a piece of input) is not the same on every occasion, depending on its current “state”.
6  - For example, in the case of a Vending machine, it will not despense a chocolate bar when you make a selection unless you have already inserted the correct amount of money. Thus, the response to the selection button depends on the previous history of the use of the system.
7 
8 ## Why FSM in Robotics
9 
10 - Robot behaviour is inherently reactive. Response to an event at any moment, is dependent on the “state” of the robot.
11 - Safety: To avoid states (motion or behaviour) with disastrous consequences, FSM only allow pre-determined state transitions.
12  - Never begin a motion unless in a safe starting position.
13  - Never keep rocket engines on for more than n seconds.
14  - (Therac-25 accident)[An investigation of the Therac-25 accidents - IEEE Journals & Magazine](https://ieeexplore.ieee.org/document/274940)

16 ## How to Structure State Machine Code
17 
18 CORC provides a structured way to build event driven FSMs. The following explains how to represent states and events and how to put them together into a CORC state machine. It is best practice to start with a state transition diagram and build each component individually. We will use the ExoTestMachine example provided in the CORC library [link](https://github.com/UniMelbHumanRoboticsLab/CANOpenRobotController/blob/master/src/apps/stateMachine/ExoTestMachine.cpp) presented bellow.

20 ![exoTestMachine](https://exoembedded.readthedocs.io/en/latest/img/exoTestMachine.png)
21 
22 - An Input Device (e.g. keyboard, controller etc) triggers Events. The desired behaviour is to move the exoskeleton between a sitting position and standing position. Using two moving states (sittingDwn and standingUp. We don't care how those motions happen, only the separation of individual State. This both seperates programatically and prevents unwanted behaviours.

24 ## States
25 
26 - States can either have an associated action with them (sitting Down), or no action (sitting), which differ them from each other.
27 - Programatically we design each state as a separate class which must implement the following three functions.
28 - The presiding Statemachine calls entry, exit and during their existence. In this view a state can be thought of as the description and behaviour of a system that is waiting to execute a transition.
29 
30 ```C++
31  virtual void entry(void);
32  -> 1 time set ups
33  virtual void durring(void);
34  -> Called by update method of the state machine
35  -> *CONTINUOUS UPDATE AND CONTROL LIVES HERE*
36  *MOTOR CONTROL*
37  virtual void exit(void);
38  -> transition out of the state triggers this, release resources.
39 ```
40

57 ## Events
58 
59 Events could a sensor trigger, timer finishing, end of a trajectory or anything in between.
60 When an event happens:
61 
62 - The event is checked against the **current state**’s transitions.
63 - If a transition matches the event, that transition “happens”.
64 - By virtue of a transition “happening”, states are **exited**, and **entered** and the relevant actions are performed
65 - The machine immediately **is in** the new state, ready to process the next event.
66 - CORC represents events as single method objects comprising as Boolean functions `virtual Boolean method: check()`. For example:
67 
68 ```C++
69 bool ExoTestMachine::EndTraj::check() {
70  return OWNER->trajectoryGenerator->isTrajectoryFinished();
71 }
72 bool ExoTestMachine::IsAPressed::check(void) {
73  if (OWNER->robot->keyboard.getA() == true) {
74  return true;
75  }
76  return false;
77 }
78 ```

80 ## Transitions
81 
82 A transition represents a shift from one State to another, triggered by a specific Event (arrow in the above diagram).
83 
84 - Transition objects have a pointer to the Event and the toState, which for a transition object is called a target. Simply a target State triggered by an Event.
85 - Programatically this is captured by the fromState holding all possible transition objects in an archList. A list of all possible transitions this state can transition too. - The overlaying State machine asks the current state to check its Archlist for any Triggered transitions. If a transition event is triggered, the transition object provides the stjatemachine a pointer to its ToState. This now becomes the current state.
86  Define each transition by specifying :
87  - FromState - the starting state for this transition
88  - ToState - the end state for this transition
89  - Event( Condition) - a callable which when it returns True means this transition is valid
90  - Use `NewTransition(_from_, _event_, _to_)` MACRO to achieve this.
91 
92 ```C++
93 \*Define sitting transitions*\
94  NewTransition(initState, startExo, sitting);
95  NewTransition(sitting, startStand, standingUp);
96 ```

98 ## Putting everything together in a stateMachine class.
99 
100 The base state machine class relies on cyclic calls to update(). - Each statemachine starts with a call to initialize(state) setting the currentState pointer to the fed in state object, - After the initialization call each loop of a program must cyclically call statemachine.update() and a program will transition from state to state based on the events which occur. - The update() function achieves this with two calls - Checks the currentStates ArcList for any triggered Transitions (that transition objects Event). - If an Event has been triggered, a sequence of calls exits the current state and sets the target to the current state. - Else if no Event triggers a transition, the currentStates during function is called. - This sequence is repeated until program exit.
101 
102 ```C++
103 void StateMachine::update()
104 {
105  Transition *t = currentState->getActiveArc();
106  if (t != NULL)
107  {
108  currentState->exit();
109  currentState = t->target;
110  currentState->entry();
111  }
112  currentState->during();
113 }
114 ```

116 ## Designing your own stateMachine
117 
118 Following the ExoTestMachine example [link](https://github.com/UniMelbHumanRoboticsLab/CANOpenRobotController/blob/master/src/apps/stateMachine/ExoTestMachine.cpp) , the following steps should yield a functional state machine.
119 
120 1. Design a state diagram with states, events and transitions (states may have more than one possible transition)
121 2. Create each state classes three virtual functions following the above SittingDwn.cpp as example
122 3. In your Statemachine class create pointers to state objects and populate with objects: `InitState *initState;`
123  &`initState = new InitState(*this*, robot, trajectoryGenerator);`
124 4. Use the EventObject(_name_) MACRO to create and initialize Event objects `EventObject(IsAPressed) * isAPressed;` & `isAPressed = new IsAPressed(*this*);`
125 5. Define Event object check() functions as above (in Statemachine class)
126 6. Pass your first state to the Base StateMachine `initialize(state)` function -> `StateMachine::initialize(initState)`
127 7. In your main program cyclically call `<yourStaeMachine>.update()`
